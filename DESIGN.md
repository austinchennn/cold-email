# 技术选型思考

> 记录每个技术决策背后的原因。不是"它能做什么"，而是"为什么选它而不是别的"。

---

## LLM：OpenAI API（兼容 Gemini）

选 OpenAI API 而不是直接集成多个 SDK，核心原因是**接口标准化**。OpenAI 的 Chat Completions 格式已经成为事实上的行业标准，Gemini、Mistral、Together AI 等都提供兼容接口。这意味着整个项目只需要维护一套调用逻辑，切换模型只改一个环境变量，不动任何业务代码。

另一个考量是 `json_mode`。这个 pipeline 大量依赖 LLM 返回结构化 JSON（教授列表、调研档案、意图识别结果），GPT-4o 的 JSON mode 相比 prompt engineering 要稳定得多——不会出现 markdown 代码块包裹、多余注释、格式漂移等问题。早期版本用纯 prompt 控制输出格式，调试成本很高，换成 json_mode 之后这类问题基本消失了。

选 GPT-4o 而不是更小的模型，是因为调研阶段需要对学术页面做深度理解和信息抽取，小模型在这类任务上的幻觉率明显更高，一旦抽取的教授信息失真，后续简历和邮件质量就会崩掉。用更强的模型保证数据质量，比用便宜模型然后加一堆校验逻辑更简洁。

## 网络搜索：Tavily API + DuckDuckGo fallback

搜索这里有一个根本问题：直接 scrape Google/Bing 会被反爬封掉，而且维护成本极高。

Tavily 是专门面向 AI agent 的搜索 API，返回的结果已经是干净的文本摘要，不需要再解析 HTML。对于这个项目来说价值很明显——Agent 1 和 Agent 2 需要的是教授研究方向的语义信息，不需要完整网页。Tavily 的结果质量比 DuckDuckGo 高，尤其是学术相关的长尾查询。

但 Tavily 是付费的，所以必须有 fallback。DuckDuckGo 有非官方 Python 包，免费且不需要 API key，作为降级方案足够用。这个"主力 + fallback"的设计让项目对没有 Tavily key 的用户也能跑起来，降低了使用门槛。

## 网页抓取：requests + BeautifulSoup

考虑过 Playwright / Selenium，但这两个是重方案：需要安装浏览器驱动、启动速度慢、资源占用高。而教授的实验室主页大多是静态或服务端渲染的，直接 HTTP 请求就能拿到内容。

BeautifulSoup 的优势是足够简单。这里的需求只是"去掉 nav、footer、script、style，留下正文文本"，不需要 DOM 操作和 JS 执行。引入 Playwright 反而是过度工程化。

真正动态渲染的页面（比如需要 JS 的 SPA）确实会抓取失败，但这种情况发生时会自动回退到搜索 snippet，整体鲁棒性可以接受。

## NLP：scikit-learn（TF-IDF + cosine similarity）

为什么不用向量数据库（Pinecone、Chroma）或语义嵌入模型？

项目池本身很小——通常只有 3-10 个项目。用向量数据库是杀鸡用牛刀，引入了外部依赖和网络调用，却没有实质收益。

TF-IDF 在这个场景下有一个被低估的优势：**可解释性**。它基于词频统计，当匹配结果不符合预期时，很容易 debug——哪些词权重高，哪些查询词命中了哪些项目，一目了然。语义嵌入是黑盒，出了问题很难定位。

另外，TF-IDF 对关键词匹配更精确。如果教授研究"transformer architecture"，TF-IDF 会直接命中包含这个词的项目；而语义嵌入有时会过度泛化，把相关性模糊的内容也匹配进来。对于简历项目这种需要精确匹配的场景，反而不合适。

## 邮件发送：Gmail API + OAuth 2.0

最简单的方案是 SMTP（smtplib）。但 SMTP 用于自动化发送容易触发 Google 的安全策略（"不够安全的应用"），而且 2022 年后 Google 已经对部分账户禁用了 SMTP 密码登录。

Gmail API 用 OAuth 2.0，是 Google 自己推荐的现代方案，稳定性和可靠性更高。另外 API 返回的 `messageId` 可以用于后续追踪（是否已回复等），SMTP 没有这个能力。

权衡是配置更复杂——需要 Google Cloud 项目和 OAuth 流程。但对于一个会实际发送邮件的工具来说，这个复杂度是值得的，总比因为 SMTP 被封掉导致发送静默失败要好。

## 重试：backoff

Gmail 发送失败有两种情况需要区分对待：
- **瞬时错误**（网络抖动、429 限速）：应该重试
- **永久错误**（401 未授权、403 禁止）：不应该重试，立刻报错

`backoff` 库的 `giveup` 参数可以优雅地表达这个逻辑，代码非常清晰。手写重试循环也能实现，但加上指数退避、最大重试次数、giveup 判断后，代码量和出错概率都会上升。

LLM 调用那边选择了手写重试循环（而不是 backoff），是因为 LLM 的重试逻辑需要带更多上下文（打日志、发事件到 bus、记录 attempt 数），用 backoff 装饰器反而会让这些逻辑分散，不如一个 for loop 直观。

## 终端 UI：Textual

最初版本是纯 `print` 输出，但当 5 个 agent 并行给同一个教授跑、每个 agent 又有多个步骤时，日志流变得很难读。

Textual 是现代 Python TUI 框架，最关键的特性是**组件化 + 异步**。每个 AgentPanel 是独立组件，状态更新不会互相干扰；`@work(thread=True)` 让 AI pipeline 在后台线程跑，UI 保持流畅响应。

对比 Rich（另一个流行选择）：Rich 擅长美化 print 输出，但它是单向的——你 print 一行，它渲染一行，没有状态管理。Textual 有真正的组件树和事件系统，更适合这种需要持续刷新多个面板的场景。

---

## 数据存储：SQLite

选 SQLite 而不是 PostgreSQL / MongoDB 的原因很简单：这个工具是单机运行的，没有并发写入需求，也不需要网络数据库服务。用 PostgreSQL 意味着用户还要额外安装和配置数据库，大幅提高了部署门槛。

SQLite 的文件即数据库，零配置，随项目目录走，备份就是复制一个文件。对于"个人使用的发送追踪"这个需求来说，完全够用。

唯一的注意点是每次操作都新建连接（而不是长连接池），这是刻意的选择——SQLite 的 WAL 模式下多连接访问没问题，而且避免了连接泄漏的隐患。

---

## 配置管理：python-dotenv

没什么特别的理由，这就是 Python 生态里的标准做法。核心价值是把密钥从代码里隔离出来，.env 文件进 .gitignore，.env.example 作为模板提交到仓库。

在 `config/settings.py` 模块顶层调用 `load_dotenv()`，所有其他模块 import settings 时就自动生效，不需要每个文件单独加载，保持了一个统一的配置入口。

## 事件总线：自研 EventBus

这里选择了手写而不是用第三方库（比如 PyDispatcher、blinker）。原因是需求非常具体：一个单例、线程安全、支持多订阅者、发布者绝不阻塞。

`queue.Queue` 是 Python 标准库里线程安全的队列，本身就是这个问题的标准解。在它基础上加一层订阅者管理，代码不超过 50 行，没有任何外部依赖，测试也很简单。

引入第三方事件库反而会带来不必要的复杂度——大多数 pub/sub 库是为回调模式设计的（事件发生时同步调用订阅者函数），但这个项目需要的是"发布者不等待消费者"的异步拉取模式，Queue 天然契合，回调模式反而需要额外处理线程安全问题。
